# CodeMosaic Brainstorm

## 1. 目标用户

- 有私有仓库但想使用外部 AI 编程助手的团队
- 对“源代码明文出境”敏感的企业研发组织
- 希望逐步引入 AI，但暂时没有条件全量自托管模型的公司

## 2. 问题拆解

真正的问题不是“是否调用 AI”，而是“如何在不暴露敏感源码的情况下仍保留足够上下文，让 AI 继续有用”。

这要求工具同时满足四件事：

1. 保留结构，不然 AI 看不懂
2. 隐藏敏感词，不然会泄密
3. 结果可回填，不然无法落地到真实仓库
4. 默认本地执行，不把映射表上传

## 3. 方案候选

### 方案 A：纯代理网关

- 优点：接入体验好，可直接包在 AI 工具前面
- 缺点：实现复杂，涉及 IDE 和网络代理协议适配

### 方案 B：离线 CLI 工作流

- 优点：最容易开源、最容易验证、部署成本低
- 缺点：需要用户手动在 masked/unmasked workspace 之间切换

### 方案 C：IDE 插件

- 优点：体验最好
- 缺点：需要先有稳定的 CLI/核心库作为基础

## 4. MVP 决策

优先做 `方案 B`，也就是：

- 本地扫描仓库
- 生成 masked workspace
- 保留本地 mapping vault
- 翻译 AI 返回 patch

这样能最快验证最核心的两个假设：

- 脱敏后 AI 仍是否有用
- patch 回填是否足够稳定

## 5. 推荐的演进路线

### Phase 1

- Python 词法级 masking
- 通用文本兜底 masking
- unified diff 回填

### Phase 2

- TypeScript/JavaScript AST 级 masking
- 更强的策略系统
- mapping 加密与审计

### Phase 3

- VS Code 扩展
- Prompt 侧最小上下文导出
- 企业策略中心与审批流

## 6. 成功指标

- 脱敏代码仍能被 AI 正确修改
- patch 回填成功率高
- 明文敏感 token 泄露率显著下降
- 用户能在 5 分钟内完成首次使用

## 7. 风险与取舍

- 脱敏越强，AI 可用性可能越差
- 结构保真越高，语义泄露概率也越高
- 纯文本兜底方案覆盖广，但精度不如 AST 方案

MVP 的合理取舍是：先做“足够好且可验证”的原型，再逐步升级到 AST 级实现。
